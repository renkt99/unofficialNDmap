# Testing (TEST) audit checklist

The single biggest testing gap in this repo is that `scripts/build-geojson.mjs`'s merge/geometry logic ‚Äî `geometryFor()` and `ringFromGeometry()` ‚Äî has zero automated coverage despite being the only code that turns raw OSM data into the polygons the map actually renders. The four browser-side modules (`js/app.js`, `js/panel.js`, `js/locate.js`, `js/search.js`) are currently untestable as written: there is no test harness in the repo at all (no `package.json`, no `node:test` suite), they're plain IIFEs coupled to `window`/`document`/Leaflet at load time, and this dev VM has no browser to exercise them in anyway. Findings from sweeps against this checklist should be logged in the audits ledger (`audits/FINDINGS.md`) under the `TEST` prefix (`TEST-1`, `TEST-2`, ...).

## Pipeline unit tests

- [ ] Add `node:test`-based tests (e.g. `scripts/build-geojson.test.mjs`, run via `node --test`) for `ringFromGeometry()`'s two branches: a way `geometry` array whose first and last points already match should come back with the same coordinate count (the `fx === lx && fy === ly` no-op path), and one whose first and last points differ must get the first point appended as a closing coordinate (the `if (fx !== lx || fy !== ly) ring.push(...)` branch). (becomes a deterministic CI gate once ci.yml is pushed)
- [ ] Test `geometryFor()` on a `relation` fixture with two or more `role: 'outer'` members that each have `geometry` ‚Äî assert it returns `{type: 'MultiPolygon', coordinates: [...]}` with one ring per outer member (the `polys.length === 1 ? Polygon : MultiPolygon` branch). (deterministic CI gate)
- [ ] Test `geometryFor()` on a `relation` fixture with no `outer`-role members carrying `geometry` ‚Äî assert it throws an `Error` matching `/relation\/\d+ has no outer members with geometry/` (the `if (!outers.length) throw` line). (deterministic CI gate)
- [ ] Test the missing-referenced-element error path in the main build loop: a curated building (`data/nd-buildings.json`) whose `osm` id is absent from `byId` (built from `footprints-raw.json`) must throw the exact `` `${b.ref}: OSM element ${b.osm} not found in footprints-raw.json` `` message. This currently lives inline in the top-level `for (const b of curated.buildings)` loop, not in a separate function, so testing it means either extracting the per-building step into a callable function or running `build-geojson.mjs` as a subprocess against a fixture `data/` directory. (deterministic CI gate, but needs a small refactor or subprocess harness first)

## validate-data.mjs gate strength

- [ ] `scripts/validate-data.mjs` has never been verified to actually fail on bad input ‚Äî as written it only ever runs against the real `data/buildings.geojson` / `data/nd-buildings.json`, so a regression that silently disables one of its checks (e.g. the `coordsWithin` bounds check or the `geoRefs.filter(...).length !== 1` uniqueness check) would go unnoticed. Add small bad-fixture files (out-of-bounds coordinate on a feature, and a duplicated/missing `ref`) under something like `scripts/fixtures/bad/` and a self-test that runs `validate-data.mjs` as a subprocess against them and asserts a non-zero exit code plus the expected error substring in stderr. (deterministic CI gate)
- [ ] Add a matching good-fixture self-test asserting the script exits `0` against the current real `data/*.json` files, so the self-test suite also guards against false positives (validate-data.mjs erroring on data that's actually fine). (deterministic CI gate)
- [ ] The campus-bounds literal (`south: -32.0615, west: 115.7405, north: -32.0515, east: 115.7515`) is duplicated between `BOUNDS` in `scripts/bounds.mjs` (imported by `scripts/validate-data.mjs` and `scripts/build-geojson.mjs`) and `CAMPUS_BOUNDS` in `js/app.js` ‚Äî kept in sync only by code comments ("Must stay in sync with..."). Add a test that imports/parses both and asserts the four numbers are identical, since nothing currently enforces this beyond a human reading the comments. (deterministic CI gate)

## Browser-code testability

- [ ] Assess extracting `js/search.js`'s `search(query)` and `buildIndex(data)` out of the enclosing IIFE into pure functions that take `index`/`data` as parameters instead of closing over the module-scoped `index` variable and `NDMap`/`escapeHtml` globals ‚Äî as written they're plain functions with no DOM reads inside `search()` itself, so the matching/ranking logic (ref-prefix tier, name-substring tier, address/contents tier, `.slice(0, 8)` cap) is close to pure already; the blocker is the lack of any export mechanism (no `type: module` / `export`, wrapped in `(function () { 'use strict'; ... })()`) and the top-level `document.getElementById(...)` calls that run at load time before `search()` is ever reached.
- [ ] `js/app.js`, `js/panel.js`, and `js/locate.js` all reference `document`, `window.NDMap`, and (for `locate.js`) `navigator.geolocation` / `L` (Leaflet) at module top level or on first call, so none of them can currently be `require`/`import`-ed under plain `node:test` without a DOM shim ‚Äî confirm whether a shim (e.g. jsdom) is wanted given the repo's no-npm-dependencies principle, or whether these three stay browser/device-only indefinitely.
- [ ] Open tradeoff, not a mandate: consider an optional headless-browser smoke test (e.g. Playwright) as a dev dependency, driving `index.html` to click a building, open the panel, run a search, and toggle the locate button ‚Äî weighed explicitly against this repo's current zero-npm-dependency, no-bundler design; if rejected, record that decision in the ledger rather than leaving it silently unconsidered.

## Manual device test protocol

- [ ] On-campus fix: with location permission granted and the device physically within `CAMPUS_BOUNDS`, tap the locate button and confirm the blue GPS dot + accuracy circle render (`drawOrUpdateDot` in `js/locate.js`) and the map pans to follow (`following` stays `true`, no "outside campus" toast). (inherently manual ‚Äî requires a real device with GPS; not automatable even once CI exists)
- [ ] Off-campus fix: with the device outside the campus bounds, confirm `onPosition()`'s `!NDMap.campusBounds.contains(latlng)` branch fires ‚Äî the "You're outside the campus area" toast appears and any existing dot is removed via `removeDot()`. (inherently manual)
- [ ] Permission-denied case: deny the browser's location permission prompt and confirm `onError()`'s `err.code === err.PERMISSION_DENIED` branch shows "Location permission denied" and calls `stopWatching()` (button reverts to its non-active state). Also exercise the generic `POSITION_UNAVAILABLE` / `TIMEOUT` toast text if the device/environment allows simulating them. (inherently manual)
- [ ] Drag-cancels-follow-mode: while watching and following, drag the map and confirm the `map.on('dragstart', ...)` handler sets `following = false` (button loses its `.following` class but keeps `.active`), then tap the locate button again and confirm it re-centers/resumes following (`if (lastLatLng && ...) map.panTo(lastLatLng)`) rather than restarting `watchPosition`. (inherently manual)

## CI wiring

- [ ] `.github/workflows/ci.yml` exists on disk (currently runs only `node scripts/validate-data.mjs` on `pull_request` and push to `main`) but is untracked in git because the `gh` token used in this environment lacks the `workflow` OAuth scope ‚Äî it has never actually been pushed/run. Push it once the token gains that scope, or have a repo admin push it manually. (this item is the prerequisite for every other "deterministic CI gate" note above)
- [ ] Once pushed, extend `ci.yml` beyond `validate-data.mjs` to also run the new pipeline `node:test` suite (e.g. add a `- run: node --test` step) so the `geometryFor()`/`ringFromGeometry()`/validate-data-self-test coverage above actually gates merges rather than being run ad hoc.
- [ ] There is no `package.json` in the repo root at all, so there's no `npm test` / `"scripts": {"test": ...}` entry point and no documented Node engine version ‚Äî add a minimal `package.json` (`"type": "module"` is implied already since the pipeline files use `.mjs`, but a `test` script standardizes how both CI and a human run `node --test` locally).
