# Correctness audit checklist (COR)

The single biggest correctness risk in this codebase is coordinate order: GeoJSON uses `[lon, lat]` while Leaflet's `L.latLng`/`L.latLngBounds` APIs use `[lat, lng]`, and the data pipeline (`scripts/build-geojson.mjs`) converts between OSM's `{lat, lon}` object form, the curated `point: [lat, lng]` convention in `data/nd-buildings.json`, and GeoJSON's `[lon, lat]` array form at three separate call sites. A single flipped conversion silently mis-places a building or POI on the map with no error anywhere in the pipeline. The second risk is that the campus lock-bounds are hard-duplicated as literal numbers in three files â€” `CAMPUS_BOUNDS` in `js/app.js`, `BOUNDS` in `scripts/validate-data.mjs`, and an inline filter in `scripts/build-geojson.mjs` â€” with only a code comment (not a shared constant or generated file) keeping them in sync; a future edit to one without the others would desync the map's pannable area from what the data pipeline accepts/produces.

Findings against this checklist go into `audits/FINDINGS.md` tagged `COR-1`, `COR-2`, etc.

## Coordinate Order & Geometry

- [ ] `scripts/build-geojson.mjs` line 59, point-building branch: `coordinates: [b.point[1], b.point[0]]`. Confirm `b.point` is always read as `[lat, lng]` per the `_comment` field in `data/nd-buildings.json` line 2, so `b.point[1]` (lng) becomes GeoJSON's `lon` slot and `b.point[0]` (lat) becomes the `lat` slot. Failure mode: any curated entry that accidentally stores `point` as `[lng, lat]` instead of `[lat, lng]` would place that building ~0.06Â° off (roughly 6-7km at this latitude) with no validation catching it, since `validate-data.mjs`'s bounds check would likely just report it as out-of-bounds rather than pinpoint the swap.
- [ ] `scripts/build-geojson.mjs` `ringFromGeometry()` (line 18-24): `geometry.map((p) => [p.lon, p.lat])` converts Overpass's `{lat, lon}` node objects to GeoJSON `[lon, lat]` pairs. Verify this mapping isn't accidentally reversed if the Overpass output shape ever changes (e.g. `out geom` vs `out center` produce different field names).
- [ ] `scripts/build-geojson.mjs` POI branch (line 86-96): `const lat = el.lat ?? el.center?.lat; const lon = el.lon ?? el.center?.lon; ... coordinates: [lon, lat]`. Verify `lon` is genuinely first in the emitted array â€” this is a separate code path from the building conversions above and uses differently-named locals, so a future edit to one path is not guaranteed to keep the other correct.
- [ ] `scripts/build-geojson.mjs` `geometryFor()` (line 26-38) for relations: only members with `role === 'outer'` are used; the code comment on line 31 explicitly says inner holes are ignored. Verified against actual data: `data/footprints-raw.json` relation `17256639` (ND5, "P&O Hotel", referenced from `data/nd-buildings.json` line 56) has exactly one `outer` and one `inner` member. This means ND5 currently renders as a solid polygon with its courtyard/inner ring silently dropped. Confirm this is accepted/intentional, or track it as a known rendering gap.
- [ ] `scripts/build-geojson.mjs` `ringFromGeometry()` ring-closing logic (line 20-22): compares `ring[0]` to `ring[ring.length-1]` by value (`fx !== lx || fy !== ly`) and pushes a closing point only if they differ. Verify this correctly handles both an already-closed Overpass way (first node repeated) and an open one â€” a bug here would produce an invalid (unclosed) GeoJSON polygon that some consumers reject.
- [ ] `scripts/build-geojson.mjs` line 33: `if (!outers.length) throw new Error(...)` guards relations with zero outer-with-geometry members, but there is no equivalent guard/warning when a relation *has* inner members that get dropped (per the item above) â€” confirm whether silently dropping holes without at least a console warning is acceptable for future data refreshes.

## Bounds Consistency

- [ ] Compare the three bounds definitions: `CAMPUS_BOUNDS` in `js/app.js` (lines 8-11: `L.latLngBounds([-32.0615, 115.7405], [-32.0515, 115.7515])`), `BOUNDS` in `scripts/validate-data.mjs` (line 16: `{ south: -32.0615, west: 115.7405, north: -32.0515, east: 115.7515 }`), and the inline filter in `scripts/build-geojson.mjs` (line 91: `lat < -32.0615 || lat > -32.0515 || lon < 115.7405 || lon > 115.7515`). As of this audit all three use identical numeric values â€” verify they still match on every future edit to any one of them; there is no shared constant, so a mismatch would only surface as a hard-to-diagnose behavioral difference (e.g. a POI that passes the build-time filter but fails `validate-data.mjs`, or vice versa).
- [ ] `scripts/build-geojson.mjs` line 91 drops out-of-bounds POIs with a bare `continue` and no counter/log line. Verify that a bounding-box or endpoint misconfiguration in `scripts/fetch-footprints.mjs` (which uses its own separately-hardcoded `BBOX` constant on line 14, a *fourth* copy of the same numbers) wouldn't silently zero out most POIs without anyone noticing at build time.
- [ ] `scripts/fetch-footprints.mjs` line 14: `const BBOX = '-32.0615,115.7405,-32.0515,115.7515'` â€” this is a fourth independent copy of the same bounds (in `south,west,north,east` string form for the Overpass query), separate from the three above. Confirm it stays numerically consistent with the others; a mismatch here would change which raw OSM elements are even fetched, upstream of every other check.

## Data Pipeline / build-geojson

- [ ] `data/nd-buildings.json` has 8 buildings using `point` (no `osm` footprint): ND2, ND8, ND9, ND15, ND16, ND34, ND38, ND39. Verified all 8 lie within `CAMPUS_BOUNDS`/`BOUNDS` today. Re-check this whenever a new point-only building is added â€” `validate-data.mjs` will catch out-of-bounds points, but only after `build-geojson.mjs` has already run.
- [ ] `scripts/build-geojson.mjs` line 55: `if (!el) throw new Error(...)` when a curated `osm` id isn't found in `footprints-raw.json`. Verify this hard-fails the build (rather than skipping the building) so a stale `footprints-raw.json` snapshot can't silently drop a building from the map.
- [ ] `scripts/build-geojson.mjs` `geometryFor()` line 32: `.filter((m) => m.role === 'outer' && m.geometry)` silently excludes outer members that lack a `geometry` field (e.g. if Overpass returns a member without `out geom` resolving it). Combined with the `!outers.length` throw on line 33, a relation with some but not all outer geometries present would build a partial (not obviously wrong) polygon rather than fail loudly.

## Search

- [ ] `js/search.js` `search()` (line 44-69): ref matching strips a leading `nd` via `refLower.replace(/^nd/, '')` (line 56) before prefix-testing against the query. Verify a query like `"25"` matches building ref `ND25` (via the stripped `refNum` path) and that a query like `"nd25"` also still matches (via the unstripped `refLower` path) â€” both `indexOf(q) === 0` checks must independently succeed for their respective cases.
- [ ] `js/search.js` line 139-141: pressing Enter calls `selectResult(currentResults[0])` â€” verify `currentResults` is always kept in sync with what's currently rendered in `resultsEl` (i.e. `renderResults` and the `currentResults` assignment on line 123 always happen together), since a desync would make Enter select something not shown on screen.
- [ ] `js/search.js` debounce (line 119-126): `search()` runs synchronously against the in-memory `index` array (built once from `ndmap:buildings-ready`, line 40-42), and `clearTimeout(debounceTimer)` unconditionally cancels any pending call before scheduling a new one. Confirm there is currently no stale-result race (there shouldn't be, since nothing async happens between keystrokes) â€” but re-verify this item if `search()` or `buildIndex()` is ever changed to do async work (e.g. a remote/lazy-loaded index), since the current guard only works because everything is synchronous.

## Locate / Geolocation State Machine

- [ ] `js/locate.js` line 177-182: `map.on('dragstart', ...)` sets `following = false` only when `watching && following` are both true. Verify a drag while `watching` is true but `following` is already false (user already panned away) is a no-op, and that dragging while not watching at all does nothing â€” both should leave `watching`/`following` unchanged.
- [ ] `js/locate.js` button click handler (line 162-174): three-way branch â€” not watching â†’ `startWatching()`; watching but not following â†’ recenter via `map.panTo(lastLatLng)` and set `following = true`; watching and following â†’ `stopWatching()`. Verify the recenter branch's guard `if (lastLatLng && NDMap.campusBounds.contains(lastLatLng))` (line 168) correctly no-ops (rather than panning to a stale/out-of-bounds point) when the last known fix was outside campus bounds.
- [ ] `js/locate.js` `onError()` (line 118-131): only `PERMISSION_DENIED` calls `stopWatching()` (line 128-130); `POSITION_UNAVAILABLE` and `TIMEOUT` just show a toast and leave `watching`/`watchId` unchanged. Verify this is intentional (transient GPS errors shouldn't kill the watch), and that the blue dot doesn't end up in a visually "stopped" state (button no longer showing `active`) while `watchId` is still a live subscription in this non-denied-error case.
- [ ] `js/locate.js` `onPosition()` (line 99-116): when a fix falls outside `NDMap.campusBounds`, it shows a toast and calls `removeDot()` but does **not** change `watching`/`following` state (line 103-108). Verify the button visual (`updateButtonVisual()`, only called from `startWatching`/`stopWatching`/the click handler) doesn't end up showing "active/following" while the dot is actually removed from the map â€” i.e. confirm the button state and dot-presence can't visibly disagree.

- [ ] `js/app.js` line 235-238: building layer clicks call `L.DomEvent.stopPropagation(e)` before opening the panel, preventing the click from also triggering the map's own `click` handler (line 153-155, which calls `NDMap.closePanel()`). - [ ] `js/panel.js` `NDMap.openPanel`/`NDMap.closePanel` (line 45-61) and `NDMap.highlightBuilding`/`NDMap.clearHighlight` in `js/app.js` (line 108-128): verify `NDMap.selectedRef` can never point at a ref whose panel is closed, or vice versa â€” i.e. every code path that opens the panel also highlights (`openPanel` line 52), and every path that closes it also clears the highlight (`closePanel` line 60), with no path that does one without the other (check `js/search.js` `selectResult()` line 98-117, which calls `NDMap.openPanel` directly, for the same invariant).

## What `scripts/validate-data.mjs` already covers, and its CI status

`scripts/validate-data.mjs` deterministically checks: both `data/buildings.geojson` and `data/pois.geojson` parse as well-formed `FeatureCollection`s with a `geometry.coordinates` on every feature (line 38-45); every coordinate pair in every feature (buildings *and* POIs, via the recursive `coordsWithin`/`walk` on line 20-27) falls within the same `BOUNDS` constant described above; every curated ref in `data/nd-buildings.json` appears in `buildings.geojson` exactly once (line 47-52); and every building feature has a non-empty `name`, a non-empty `contents` array, and a `confidence` value in `{high, medium, low}` (line 53-58). It does **not** validate anything about POI properties (`kind`/`name` on POI features are never checked for presence or validity), does not check ring-closure or winding order of polygons, and does not detect the dropped-inner-ring issue described above (a solid polygon is still a valid, in-bounds polygon).

A `.github/workflows/ci.yml` file exists in the working tree and, if committed, would run `node scripts/validate-data.mjs` on every `pull_request` and push to `main` (line 9-17). It is currently **untracked** (per this repo's own `CLAUDE.md`: "there is no CI gate yet â€” see BLD-001 in `audits/FINDINGS.md`"), so GitHub Actions does not execute it yet â€” the gate is manual-only today, run by convention before every merge, not enforced. Do not describe this workflow as "wired into CI" until it is actually committed/pushed and confirmed running on a real PR.

- [ ] Confirm `.github/workflows/ci.yml` stays accurate (currently: runs `node scripts/validate-data.mjs` on `pull_request` and push to `main`, line 9-17) and get it committed/tracked so the gate is actually enforced (see BLD-001) â€” until then, treat `validate-data.mjs` as a manual pre-merge step only. Separately, note that even once active, it only validates whatever `data/buildings.geojson`/`data/pois.geojson` are already committed â€” it does not itself run `scripts/build-geojson.mjs`, so a stale committed `.geojson` that no longer matches `data/nd-buildings.json` / `data/footprints-raw.json` would pass CI even if regenerating it would change the output or fail.
